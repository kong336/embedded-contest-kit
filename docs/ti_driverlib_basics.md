# TI DriverLib Basics

This page turns the TI software stack into a practical cheat sheet.

## First thing to know

TI does not use one universal "HAL" name across all MCU lines.

What you will actually see is:

- `MSPM0-SDK` with `DriverLib`
- `MSP430Ware` with `DriverLib`
- `C2000Ware` with `DriverLib`
- `TivaWare` for `TM4C`

So if you search for "TI HAL", the practical answer is usually:
start from the family-specific DriverLib or TivaWare peripheral library.

## Family mapping

| Family | Official software package | Common API style |
| --- | --- | --- |
| `MSPM0` | `MSPM0-SDK` | `DL_GPIO_*`, `DL_UART_Main_*`, `DL_Timer_*`, `DL_ADC12_*` |
| `MSP430` | `MSP430Ware` | `GPIO_*`, `EUSCI_A_UART_*`, `Timer_A_*`, `ADC_*` |
| `C2000` | `C2000Ware` | `GPIO_*`, `SCI_*`, `EPWM_*`, `ADC_*` |
| `TM4C` | `TivaWare` | `GPIOPin*`, `UART*`, `PWM*`, `ADC*`, `SysCtl*` |

## The minimum bringup path

1. Install the family software package.
2. Open a vendor example for your exact LaunchPad or MCU.
3. Let the example or SysConfig output own clocks, pinmux, and low-level init.
4. Add only a thin wrapper layer for the operations you reuse all the time.

That is the purpose of the `ti_driverlib/` folder in this repository.

## Basic operation map

### GPIO output

| Family | Common functions |
| --- | --- |
| `MSPM0` | `DL_GPIO_enableOutput`, `DL_GPIO_setPins`, `DL_GPIO_clearPins`, `DL_GPIO_togglePins` |
| `MSP430` | `GPIO_setAsOutputPin`, `GPIO_setOutputHighOnPin`, `GPIO_setOutputLowOnPin`, `GPIO_toggleOutputOnPin` |
| `C2000` | `GPIO_setDirectionMode`, `GPIO_writePin`, `GPIO_togglePin` |
| `TM4C` | `GPIOPinTypeGPIOOutput`, `GPIOPinWrite`, `GPIOPinRead` |

### UART transmit

| Family | Common functions |
| --- | --- |
| `MSPM0` | `DL_UART_Main_transmitDataBlocking` |
| `MSP430` | `EUSCI_A_UART_transmitData` |
| `C2000` | `SCI_writeCharBlockingFIFO` |
| `TM4C` | `UARTCharPut` |

### PWM update

| Family | Common functions |
| --- | --- |
| `MSPM0` | `DL_Timer_setCaptureCompareValue` |
| `MSP430` | `Timer_A_setCompareValue` |
| `C2000` | `EPWM_setCounterCompareValue` |
| `TM4C` | `PWMPulseWidthSet` |

### ADC read

| Family | Common functions |
| --- | --- |
| `MSPM0` | `DL_ADC12_startConversion`, `DL_ADC12_getMemResult` |
| `MSP430` | `ADC_startConversion`, `ADC_getResults` |
| `C2000` | `ADC_forceSOC`, `ADC_readResult` |
| `TM4C` | `ADCProcessorTrigger`, `ADCSequenceDataGet` |

## What belongs in the reusable layer

Good candidates:

- LED or status pin wrappers
- UART send helpers
- PWM compare update helpers
- blocking single-channel ADC helpers for quick tests

Keep outside the reusable layer:

- board-specific clock trees
- pinmux code generated by SysConfig
- one-board-only launch examples
- every single peripheral init parameter

## Repository starter headers

These files are thin helpers, not full SDK replacements:

- `ti_driverlib/mspm0/contest_ti_mspm0_driverlib.h`
- `ti_driverlib/msp430/contest_ti_msp430_driverlib.h`
- `ti_driverlib/c2000/contest_ti_c2000_driverlib.h`
- `ti_driverlib/tm4c/contest_ti_tm4c_driverlib.h`

They are meant to save you from rewriting the same tiny glue every time you start a new TI-side test project.

## Important limitation

These headers were added as source templates and were not compiled in this repository, because the TI SDKs are not bundled in the current workspace.

That means:

- the API direction and names follow official TI documentation
- you should still verify include paths, generated init code, and exact macro names against your installed SDK version
